# VoicePaste 语料库与分析进度板

最后更新：2026-06-13

这个文件是 VoicePaste「语料库 / 原始识别文本 / 历史记录 / 导出 / 分析」需求的协作进度板。后续不管是 Codex、Claude，还是其他 agent 接手，都必须先看这里，避免只靠聊天记录续上下文。

任何 agent 完成、跳过、阻塞、重新定义下面任意事项后，都必须在最终回复前更新这个文件。

## 0. 当前总原则

- 日常语音输入必须保持快。语料记录和分析功能不能让「按快捷键 -> 说话 -> 粘贴」产生明显延迟。
- 每次语音只写入一条 JSONL 历史记录，不要为了 raw/final 拆成多条记录或多个文件。
- 历史写入必须放在粘贴成功之后；如果后续发现同步写入有风险，再进一步改成延后写入。
- AI 分析必须手动触发、分块处理、可续跑、可缓存。禁止把长期语料一次性塞进一个 prompt。
- 浮层窗口只负责录音反馈，保持轻量；语料浏览、导出、分析都放在设置窗口里。
- 开始做语料库、历史记录、导出、分析相关任务前，必须先读这个文件。
- 做完相关任务后，必须先更新这个文件，再给用户最终回复。

## 2. Phase 1 - 先保住 raw 语料数据

目标：从新录音开始，不再丢失豆包原始识别文本，同时不改变用户的语音输入手感。

- [x] 在豆包 ASR final 文本确定后、进入 LLM 润色或其他最终文本处理前，捕获 `rawText`。
- [x] 把真正粘贴出去的文本保存为 `finalText`。
- [x] 保留旧字段 `text`，并让它等于 `finalText`，保证旧 UI 和旧统计逻辑不坏。
- [x] 新增元数据字段：`mode`、`promptId`、`llmProvider`、`llmModel`、`durationMs`、`chars`、`ts`。
- [x] 每次语音只追加一条 JSONL 记录，raw/final 都在同一条记录里。
- [x] 确保历史写入仍然发生在粘贴成功之后。
- [x] 如果同步写入有可测延迟或风险，把历史写入改为延后执行，避免阻塞热键和粘贴流程。
- [x] 兼容旧历史记录：旧记录没有 `rawText` 时，读取、展示、统计都要正常。
- [x] 代码改动后运行 `pnpm check`。

验证记录（2026-06-13）：

- `pnpm check` 通过。
- `node scripts/inspectLatestHistory.js` 通过；当前最新磁盘记录仍是旧 schema，脚本能派生 `rawText/finalText` 并显示 `storageSchemaComplete: false`。
- 待用户录入新的普通语音和润色语音后，再运行 `node scripts/inspectLatestHistory.js` 复核新记录应显示 `storageSchemaComplete: true`，并确认 `mode=normal/polish`。

## 3. Phase 2 - 语料库前端 UI

目标：在设置窗口里增加一个清晰入口，用来浏览、筛选、导出语料。

推荐导航位置：

- 在左侧导航「概览」下面新增 `语料库`。
- `首页` 继续只做快速概览和最近输入。
- `使用统计` 继续只做累计统计和图表。
- 不要把语料库控制放到录音浮层里。

推荐 `语料库` 页面分成 3 个页签：

- [ ] `记录`：查看、搜索、筛选 raw/final 历史记录。
- [ ] `导出`：按日期、模式、模板导出语料。
- [ ] `分析`：手动发起分析，查看缓存的分析结果。

推荐 `记录` 页先做这些控件：

- [ ] 日期范围：今天、近 7 天、近 30 天、自定义。
- [ ] 模式筛选：全部、普通识别、润色。
- [ ] 搜索框：同时搜索 raw 和 final 文本。
- [ ] 每条记录显示：时间、录音时长、字数、模式、prompt。
- [ ] 默认只显示 `finalText`。
- [ ] 展开后显示 `rawText`、`finalText` 和完整元数据。
- [ ] 操作按钮：复制 raw、复制 final、删除。

## 4. Phase 3 - 本地统计

目标：先做不用 AI 的分析，速度快、成本低，也能给后续 AI 分析提供基础数据。

- [ ] 按天 / 周 / 月统计输入量。
- [ ] 统计录音时长、字数、类似「每分钟输入字数」的效率指标。
- [ ] 统计高频输入时段。
- [ ] 统计高频词和可能需要加入热词表的专有词。
- [ ] 检测基础口语习惯：重复词、填充词、常见口头禅。
- [ ] 在同时有 raw/final 的记录里，统计润色改动幅度。

## 5. Phase 4 - AI 分析

目标：做更深的语料分析，但不能压垮模型上下文，也不能影响正常语音输入。

- [ ] 提供手动「运行分析」按钮。
- [ ] 运行前允许选择日期范围和模式。
- [ ] 长文本按日期和大小分块。
- [ ] 缓存每天 / 每块的分析结果。
- [ ] 中断后能从最后完成的块继续。
- [ ] 显示进度状态：排队中、运行中、已完成、失败。
- [ ] 禁止把全部长期语料一次性发送给 AI。

第一批推荐的 AI 分析报告：

- [ ] 高频词和个人词库候选。
- [ ] ASR 可能误识别模式。
- [ ] 口语习惯：重复表达、长句、填充词。
- [ ] 主题分布和主题迁移。
- [ ] 润色收益：raw 到 final 到底改了什么。
- [ ] 输入效率：高产日期、高产时间段、输入节奏。

## 7. 下一步推荐顺序

1. 先做 Phase 1，因为现在每新增一次语音输入，都会继续丢失独立的 raw/final 信息。
2. 再做 Phase 2 的 `语料库` 页面骨架，先有 `记录 / 导出 / 分析` 三个页签；导出和分析可以先只放入口或空状态。
3. 再做 Phase 3 本地统计，因为它便宜、快、稳定，也能为后续 AI 分析打基础。
4. 最后做 Phase 4 AI 分析，必须走手动、分块、缓存、可续跑。
